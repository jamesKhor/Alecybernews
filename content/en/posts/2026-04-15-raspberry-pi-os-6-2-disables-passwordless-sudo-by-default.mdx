## Executive Summary
Raspberry Pi OS 6.2, released on April 15, 2026, has disabled passwordless `sudo` by default for new installations. This change, a significant departure from the platform's long-standing default behavior, is a deliberate security hardening measure designed to prevent unauthorized privilege escalation on the ubiquitous single-board computers. The update forces users to enter their account password when executing commands with superuser privileges, closing a common attack vector for malware and unauthorized local access.

## Technical Analysis
The core change in Raspberry Pi OS 6.2, based on Debian 13 "Trixie," modifies the configuration of the `sudo` command. Previously, the default user (`pi`) was a member of the `sudo` group and could execute commands with root privileges without re-entering a password. According to the Raspberry Pi Foundation, this configuration has been removed from the standard `sudoers` file for new installations. The change is implemented by ensuring the default user is not granted the `NOPASSWD` tag. For existing installations upgrading to version 6.2, the configuration remains unchanged to preserve system stability and user experience, though administrators are advised to review their `sudoers` policy. This move aligns Raspberry Pi OS more closely with standard Debian security practices, where passwordless sudo is not a default.

## Indicators of Compromise
None identified at this time. This is a proactive configuration change, not a response to a specific active exploit.

## Tactics, Techniques & Procedures
The default configuration of passwordless `sudo` presented a clear security risk that could be leveraged in several attack scenarios. An adversary with initial access to a user's session—whether through a compromised application, a malicious script, or physical access to an unlocked device—could immediately escalate to root privileges without needing to brute-force or steal a password. This technique, often categorized under Privilege Escalation (T1548), lowers the barrier for post-exploitation activities such as installing persistent malware, disabling security controls, or accessing protected system files. The change mitigates this by enforcing a requirement for re-authentication (T1550.002 - Use Alternate Authentication Material: Password).

## Threat Actor Context
While not linked to a specific threat actor or campaign, the risk of passwordless sudo has been a well-known concern in security communities for years. Raspberry Pi devices are deployed in a wide range of environments, from educational labs and IoT prototypes to industrial control systems and home servers. Their low cost and accessibility make them attractive targets for botnet herders seeking to conscript devices into DDoS networks or for attackers looking for a foothold in a network. The default security posture has historically prioritized ease of use for beginners, a decision the Foundation now acknowledges requires continual review and adjustment against evolving threat landscapes.

## Mitigations & Recommendations
*   **New Installations:** Users performing fresh installs of Raspberry Pi OS 6.2 will automatically benefit from the new, more secure default. No action is required.
*   **Existing Systems:** Administrators of existing Raspberry Pi devices should manually review and update their sudo configuration. This can be done by editing the `/etc/sudoers.d/010_pi-nopasswd` file (if present) and removing the `NOPASSWD:` tag, or by using the `visudo` command to edit the main `sudoers` file. The recommended line for the `pi` user should be: `pi ALL=(ALL) ALL`.
*   **General Hardening:** This change should be part of a broader hardening strategy for internet-facing Pis, which should also include changing the default `pi` username, disabling password-based SSH authentication in favor of key-based auth, and ensuring regular system updates.
*   **Automation Scripts:** Users and developers relying on automated scripts that use `sudo` will need to adjust their workflows, potentially using password-less SSH keys for specific automated tasks or configuring `sudo` with tailored, secure policies for specific commands.
