## 执行摘要
Google 已经开始在 Windows 系统上的 Chrome 146 版本中分阶段推出一个关键的防御机制，旨在中和信息窃取恶意软件的主要攻击途径。新特性 **设备绑定会话凭证（DBSC）** 通过将会话 cookie 加密绑定到原始设备的硬件上，使被盗的 cookie 在其他机器上用于账户接管变得无效。这一变化直接针对恶意软件家族如 **RedLine**、**Vidar** 和 **Lumma** 对认证 cookie 的猖獗盗窃，这助长了被泄露账户的庞大地下经济。

## 技术分析
DBSC 保护增强了现有的 `__Host-` 安全 cookie 前缀，增加了一个设备绑定的密钥组件。根据 Google 的实现细节，当用户登录支持的网站时，Chrome 会生成一个与设备的信任平台模块（TPM）或（如果不可用）设备硬件本身相关联的唯一私钥。此密钥用于签名会话 cookie。相应的公钥发送到网站的服务器。当浏览器随后为认证呈现 cookie 时，它还必须通过加密签名证明私钥的拥有权。被盗的 cookie 文件缺少这些密钥材料，使得从不同设备进行认证变得不可能。

Google 的推广是谨慎的，需要网站明确采用。该特性最初仅对 Windows 上登录浏览器的一小部分 Chrome 146 用户启用。网站必须通过设置新的 `DeviceBoundSessionCredential` cookie 属性来选择加入。由于其强大的硬件支持安全堆栈（TPM），保护目前仅限于 Windows，但 Google 已表示计划将其扩展到其他桌面平台。该机制的有效性取决于主要在线服务的广泛采用。

## 入侵指标
目前没有识别出任何指标。DBSC 是一种主动缓解措施，不是对特定新活动的反应。

## 战术、技术与程序
DBSC 特性直接对抗已建立的对手技术。信息窃取恶意软件通常采用以下 TTP 来利用传统 cookie：
*   **T1555.003: 从 Web 浏览器获取凭证** —— 恶意软件从磁盘上刮取浏览器 cookie 数据库（例如 `Cookies` SQLite 文件）。
*   **T1539: 窃取 Web 会话 Cookie** —— 被盗的 cookie 文件被泄露到攻击者控制的服务器。
*   **T1606: 伪造 Web 凭证** —— 攻击者或其客户将这些 cookie 载入单独的浏览器（通常使用专业工具）以冒充受害者并绕过密码和多因素认证（MFA）等认证机制。

通过将会话绑定到硬件，DBSC 打破了 `T1606: 伪造 Web 凭证` 步骤，因为攻击者无法从不同设备产生所需的加密证明。

## 威胁行为者背景
会话 cookie 盗窃的主要受益者是信息窃取恶意软件（infostealers）的运营商和购买这些被盗凭证的犯罪地下买家。Infostealers 通常通过钓鱼、恶意广告、盗版软件和假冒下载站点进行分发。被盗的 cookie 被打包成“日志”并在 Telegram 频道和暗网市场上出售。买家使用它们进行金融欺诈、企业间谍活动、社交媒体账户接管和进一步的网络入侵。这一生态系统使 cookie 盗窃成为一种低成本、高回报的攻击，对个人和组织造成了毁灭性的影响。DBSC 旨在降低这种商品的价值。

## 缓解措施与建议
*   **对于最终用户和企业：** 确保 Windows 端点上的 Chrome 更新到 146 版本或更高版本。在设备固件中启用 TPM 等硬件安全功能，为 DBSC 提供最强大的基础。
*   **对于网站运营商和安全团队：** 积极实施对设备绑定会话凭证标准的支持。Google 提供了集成的开发人员文档。优先考虑为高价值账户（行政、财务、电子邮件）推广。这应该是与强大的端点检测和响应（EDR）一起作为深度防御策略的一部分，以在 infostealers 执行之前捕获它们。
*   **一般安全姿态：** DBSC 不能阻止恶意软件从浏览器窃取密码、自动填充数据或其他敏感信息。它不是全面反恶意软件工具、用户安全培训和通过账户安全设置及时撤销可疑会话的替代品。
